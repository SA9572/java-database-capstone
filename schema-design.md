# Smart Clinic Management System — Schema Design

## MySQL Database Design

Relational, structured data — patients, doctors, appointments, and admin accounts — lives in MySQL because these entities have clear relationships (foreign keys) and need strong consistency (e.g., an appointment must always reference a valid doctor and patient).

### Table: patients
- id: INT, Primary Key, Auto Increment
- name: VARCHAR(100), Not Null
- email: VARCHAR(100), Not Null, Unique
- password: VARCHAR(255), Not Null
- phone: VARCHAR(10), Not Null
- address: VARCHAR(255), Not Null

> `email` is marked UNIQUE to prevent duplicate patient registrations. Password format/strength is validated in code (`@Size(min = 6)`), not enforced at the DB level.

### Table: doctors
- id: INT, Primary Key, Auto Increment
- name: VARCHAR(100), Not Null
- specialty: VARCHAR(50), Not Null
- email: VARCHAR(100), Not Null, Unique
- password: VARCHAR(255), Not Null
- phone: VARCHAR(10), Not Null

> Doctor `availableTimes` is stored in a **separate table** (`doctor_available_times`) since it's a one-to-many list of strings, not a single column — this is handled automatically by JPA's `@ElementCollection`.

### Table: doctor_available_times
- doctor_id: INT, Foreign Key → doctors(id)
- available_time: VARCHAR(20) (e.g., "09:00-10:00")

> This table is auto-generated by `@ElementCollection` on the `Doctor` entity — no manual model class needed for it.

### Table: appointments
- id: INT, Primary Key, Auto Increment
- doctor_id: INT, Foreign Key → doctors(id), Not Null
- patient_id: INT, Foreign Key → patients(id), Not Null
- appointment_time: DATETIME, Not Null
- status: INT (0 = Scheduled, 1 = Completed, 2 = Cancelled), Not Null

> **On patient/doctor deletion:** rather than a hard `ON DELETE CASCADE` at the DB level (which could silently wipe appointment history), deletions are handled in the service layer — e.g., `DoctorService.deleteDoctor()` explicitly removes that doctor's appointments first. This keeps deletion behavior visible and controllable in application code instead of hidden in DB triggers.
>
> **Overlapping appointments:** prevented in code via `DoctorService.isDoctorAvailable()`, which checks existing appointments for that doctor on the same day before allowing a new booking — not enforced by a DB constraint, since "overlap" logic (accounting for the 1-hour appointment duration) is easier to express in Java than in SQL.

### Table: admin
- id: INT, Primary Key, Auto Increment
- username: VARCHAR(50), Not Null, Unique
- password: VARCHAR(255), Not Null

> Only one table for admin since the system has a small, fixed set of administrators — no need for additional relational complexity here.

### Design decisions, thought through
- **Should a prescription be tied to an appointment, or exist independently?** Tied to an appointment (via `appointmentId`), since a prescription only makes sense in the context of a specific consultation. This reference lives in MongoDB, not as a MySQL foreign key, since Prescription is a MongoDB document (see below) — the link is enforced at the application layer instead of the database layer.
- **Should a patient's appointment history be retained forever?** Yes — appointments are kept even after `status = 2 (Cancelled)` or `status = 1 (Completed)`, since this history is valuable for both the patient (recurring care context) and the clinic (auditing, no-show tracking). Nothing is hard-deleted except by explicit admin action.
- **Passwords:** stored as VARCHAR(255) to leave room for hashed values (e.g., BCrypt output), even though the current lab implementation stores them in plain text for simplicity. This is flagged as a follow-up hardening step, not a final design.

## MongoDB Collection Design

Free-form or evolving data — prescription details, doctor notes, and future features like patient feedback or chat logs — fits MongoDB better than rigid SQL tables, since these documents don't all share the same shape and may gain new fields over time without requiring a schema migration.

### Collection: prescriptions

```json
{
  "_id": "ObjectId('64abc123456')",
  "patientName": "John Smith",
  "appointmentId": 51,
  "medication": "Paracetamol",
  "dosage": "500mg",
  "doctorNotes": "Take 1 tablet every 6 hours after meals.",
  "refillCount": 2,
  "tags": ["fever", "follow-up"],
  "pharmacy": {
    "name": "Walgreens SF",
    "location": "Market Street"
  }
}
```

> **Why only `appointmentId` and `patientName`, not the full Patient object?** Storing just the ID keeps this document lightweight and avoids duplicating data that can change (e.g., if a patient updates their address, we don't want stale copies scattered across every prescription document). `patientName` is denormalized here only because pharmacies/doctors printing this out don't want to do a lookup back into MySQL just to display a name — a reasonable, deliberate tradeoff between normalization and read convenience.
>
> **Schema evolution:** since MongoDB is schema-less, future fields (e.g., `refillCount`, `tags`) can be added to new documents without migrating old ones — existing prescriptions simply won't have that field until updated, and application code treats it as optional.

### Example: a future `logs` collection (not yet implemented, included for completeness)

```json
{
  "_id": "ObjectId('64def789012')",
  "patientId": 12,
  "eventType": "check-in",
  "timestamp": "2026-07-02T09:15:00Z",
  "metadata": {
    "location": "Front Desk",
    "notes": "Patient arrived 10 minutes early"
  }
}
```

> Log/event data like check-ins or chat messages is a natural fit for MongoDB since each event type may carry different metadata — a rigid SQL table would need many nullable columns to accommodate every possible event shape.
